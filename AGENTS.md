# AGENTS.md - GreenSkills

## Project Overview

GreenSkills is a mobile Serious Game for CSR/ecology micro-learning. It has two components:

- **Client/** - Unity 6 (6000.3.3f1) project, C# 9.0, targeting .NET Framework 4.7.1
- **Server/** - Go 1.25.6 REST API (module: `github.com/gauthier-se/GreenSkills-API`)

## Build & Run Commands

### Unity Client

The client is a Unity project. It cannot be built from CLI without Unity installed.

```bash
# Open in Unity Hub (macOS)
open -a "Unity Hub" Client/

# CLI build (requires Unity installed at the standard path)
/Applications/Unity/Hub/Editor/6000.3.3f1/Unity.app/Contents/MacOS/Unity \
  -projectPath Client/ -buildTarget Android -batchmode -nographics -quit

# Run Unity tests (Edit Mode + Play Mode)
/Applications/Unity/Hub/Editor/6000.3.3f1/Unity.app/Contents/MacOS/Unity \
  -projectPath Client/ -runTests -batchmode -nographics -quit \
  -testResults ./test-results.xml

# Run a single test class
/Applications/Unity/Hub/Editor/6000.3.3f1/Unity.app/Contents/MacOS/Unity \
  -projectPath Client/ -runTests -batchmode -nographics -quit \
  -testFilter "ClassName"

# Run a single test method
/Applications/Unity/Hub/Editor/6000.3.3f1/Unity.app/Contents/MacOS/Unity \
  -projectPath Client/ -runTests -batchmode -nographics -quit \
  -testFilter "ClassName.MethodName"
```

No `.editorconfig`, Roslyn analyzers, or linting tools are configured. Warnings 0169 and 0649 are suppressed.

### Go Server

```bash
# Build
cd Server && go build -o bin/server ./cmd/server

# Run
cd Server && go run ./cmd/server

# Run all tests
cd Server && go test ./...

# Run a single test
cd Server && go test -run TestName ./...

# Run tests with verbose output
cd Server && go test -v ./...

# Run tests with coverage
cd Server && go test -v -cover ./...

# Run integration tests only
cd Server && go test -tags=integration ./...
```

## Git Conventions

### Commit Messages

- Written in **English**
- Use [Conventional Commits](https://www.conventionalcommits.org/) with a scope and Linear issue ID at the end
- Format: `type(scope): short description (TE-XXX)`
- Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `ci`, `style`, `perf`
- Common scopes: `api`, `client`, `db`, `auth`, `ui`
- Keep the subject line under 72 characters
- Examples:
  - `feat(api): add health check endpoint with middleware stack (TE-173)`
  - `fix(client): correct exercise type parsing for fill-in-blank (TE-150)`
  - `test(api): add integration tests for auth endpoints (TE-175)`
  - `docs: update AGENTS.md with Go conventions (TE-173)`

## C# Code Style

### Namespaces

- Mirror the folder structure: `Managers`, `Data`, `Data.Exercises`, `UI`, `UI.Exercises`, `Testing`
- No root project prefix
- Use block-scoped namespaces with Allman braces (never file-scoped `namespace X;`)

### Imports (using directives)

- Place `using` statements **outside** the namespace block
- Order: `System.*` first, then project namespaces, then `UnityEngine.*` / `TMPro` last
- No `global using`, `using static`, or aliased usings

### Naming Conventions

| Element                | Convention              | Example                                |
|------------------------|-------------------------|----------------------------------------|
| Classes                | PascalCase              | `GameManager`, `QuizController`        |
| Interfaces             | `I` + PascalCase        | `IExerciseController`                  |
| Methods                | PascalCase              | `LoadLevel()`, `ValidateAnswer()`      |
| Event handlers         | `On`/`Handle` prefix    | `OnPlayClick()`, `HandleAnswerSubmitted()` |
| Private fields         | `_camelCase`            | `_currentLevelId`, `_exerciseData`     |
| [SerializeField] fields| `camelCase` (no prefix) | `maxLives`, `feedbackDelay`            |
| Public data fields     | `camelCase`             | `exerciseType`, `questionText`         |
| Properties             | PascalCase              | `Instance`, `LevelId`, `IsUnlocked`    |
| Constants              | SCREAMING_SNAKE_CASE    | `HIGHEST_LEVEL_KEY`                    |
| Enums                  | PascalCase type+members | `ExerciseType.Quiz`, `Difficulty.Easy` |
| Local variables        | camelCase               | `levelToLoad`, `isCorrect`             |
| Parameters             | camelCase               | `apiUrl`, `exerciseData`               |

### Class Suffixes

- `Manager` for singleton/service classes: `GameManager`, `ExerciseManager`
- `Controller` for UI controllers: `QuizController`, `SortingController`
- `Data` for ScriptableObject data models: `QuizExerciseData`, `BaseExerciseData`
- `Dto` for data transfer objects: `BaseExerciseDto`, `GenericExerciseDto`

### Formatting

- **Allman braces** everywhere (opening brace on its own line)
- **4-space indentation** (no tabs)
- Space after keywords: `if (`, `for (`, `foreach (`
- Space around operators: `i + 1`, `index < 0`
- Blank line between methods and after field blocks
- Always use explicit access modifiers (`public`, `private`, `protected`)
- Prefer braces on single-line bodies (braceless form is rare)

### Modern C# Features (C# 9.0)

- Switch expressions: `state switch { MatchableItemState.Normal => normalColor, ... }`
- Tuple deconstruction for swaps: `(list[i], list[j]) = (list[j], list[i])`
- String interpolation (never `string.Format`): `$"Level {id} loaded"`
- Null-conditional: `controller?.Reset()`, `OnAnswerSubmitted?.Invoke(...)`
- Null-coalescing: `options?.Count ?? 0`
- Expression-bodied properties: `public int LevelId => _levelId;`

### Documentation

- Use XML `/// <summary>` on all public and protected members
- Include `<param>` and `<returns>` tags where applicable
- Use `[Header("Section Name")]` to group SerializeField blocks in the Inspector
- Use `[Tooltip("description")]` on data class fields
- Use `#region` / `#endregion` to organize large manager files
- Inline comments only for non-obvious logic

### Error Handling

- **No exceptions**: use `Debug.LogError` / `Debug.LogWarning` / `Debug.Log` with early returns
- Prefix log messages with `[ClassName]`: `Debug.LogError("[GameManager] ExerciseManager is null!")`
- Use guard clauses (null checks + early return) at method entry
- Try-catch only for JSON deserialization; catch `System.Exception`, log, and return null
- No custom exception types

### Unity Patterns

- **Singleton**: manual pattern with `public static T Instance { get; private set; }` in `Awake()`, with `DontDestroyOnLoad`
- **MonoBehaviour**: all manager/controller/UI classes inherit from `MonoBehaviour`
- **ScriptableObject**: all exercise data classes, with `[CreateAssetMenu]` attribute
- **Coroutines**: `StartCoroutine(MethodName())` with `yield return new WaitForSeconds(...)`
- **Async/await**: used in `ExerciseManager` for data loading (`Task<T>` return types)
- **Events**: `event Action<T>` for decoupled communication; clean up in `OnDestroy()`
- **[RequireComponent]**: used on classes that depend on specific components
- **[System.Serializable]**: on all DTO/helper data classes (full qualifier, not `using System`)
- **PlayerPrefs**: for local persistence (level progress, scores)
- **Preprocessor directives**: `#if UNITY_EDITOR` for editor-only code (e.g., `EditorApplication.isPlaying = false`)

### Architecture Patterns

- **Manager + Controller + Data** three-layer separation
- **Factory**: `ExerciseFactory` (static) converts DTOs to domain objects via switch expressions
- **Strategy**: `IExerciseController` with `Dictionary<ExerciseType, IExerciseController>` dispatch
- **Template Method**: `BaseExerciseController` defines virtual/abstract lifecycle methods
- **Observer**: C# events (`Action<T>`) for decoupled communication between layers

### File Organization

- One class per file as the default
- Related small DTOs may share a file (e.g., `ExerciseDtos.cs`)
- A closely related enum may live in the same file as its primary class
- UI-facing strings may be in French; all code/comments/logs are in English

### Architecture

- **Handler layer**: receives HTTP requests, validates input, calls services/DB, writes JSON responses
- **Database layer**: `pgxpool.Pool` passed to handlers via dependency injection (struct fields, not globals)
- **Config layer**: loads `.env` + environment variables into a typed `Config` struct
- **Middleware layer**: chi middleware functions for cross-cutting concerns
- **No globals**: all dependencies injected via struct fields or function parameters
- **No ORM**: raw SQL with `sqlc` for type-safe generated Go code

### HTTP Conventions

- All API routes prefixed with `/api`
- JSON responses with `Content-Type: application/json`
- Error responses use a consistent format: `{"error": "message"}`
- HTTP status codes: `200` success, `201` created, `400` bad request, `401` unauthorized, `404` not found, `409` conflict, `500` internal error
- Use `http.StatusXxx` constants, not raw integers

### Error Handling

- Return errors, don't panic (panics caught by recovery middleware as last resort)
- Use `fmt.Errorf("context: %w", err)` for error wrapping
- Log errors with `log/slog` (structured logging)
- Handler functions write error JSON and return early on error
- No custom error types unless truly needed

### Testing

Every endpoint must have:

1. **Unit tests** (`*_test.go` next to the handler):
   - Test handler functions in isolation
   - Use `httptest.NewRequest` + `httptest.NewRecorder`
   - Test success cases and error cases
   - Test with mock dependencies (interfaces for DB access)
   - Naming: `TestHandlerName_Scenario` (e.g., `TestHealth_ReturnsOK`, `TestLogin_InvalidPassword`)

2. **Integration tests** (`*_integration_test.go` or build-tagged):
   - Test full HTTP request/response cycle with a real or test database
   - Use `httptest.NewServer` with the real router
   - Verify actual DB interactions when applicable
   - Use `// +build integration` tag or `_integration_test.go` suffix

3. **Bruno collection** (`Server/bruno/`):
   - One `.bru` file per API request
   - Organized by resource (e.g., `health/`, `auth/`, `levels/`)
   - Use environment variables for base URL and auth tokens
   - Include example request bodies and expected responses

```bash
# Run unit tests only
cd Server && go test ./...

# Run all tests including integration
cd Server && go test -tags=integration ./...

# Run tests for a specific package
cd Server && go test ./internal/handler/...

# Run a single test
cd Server && go test -run TestHealth_ReturnsOK ./internal/handler/...

# Run tests with verbose output and coverage
cd Server && go test -v -cover ./...
```
